# MVP Essentials for Premium Site OS

Date: 2026-05-17

## Purpose

This document compresses the full Premium Site OS into the smallest practical workflow people can use immediately.

The master system can stay deep. The daily workflow should feel simple:

> Give links, images, and a business idea. Get a premium direction, build packet, and proof that it beats generic AI output.

## 1. Tiny First Workflow

Use this when speed matters.

### Inputs

Minimum:

- Business name
- Offer
- Audience
- Desired action
- One image or one URL

Better:

- 1 to 3 images
- 1 owned site URL
- 2 to 3 competitor/reference URLs
- Existing copy or notes
- Must-avoid list

### Five-Step Workflow

1. **Intake**
   Capture the business, audience, offer, goal, and source materials.

2. **Source Read**
   Classify each input as lead, support, constraint, rejection, or trend signal.

3. **Creative Direction**
   Choose premium mode, visual language, motion role, and anti-slop rules.

4. **Build Packet**
   Produce the art bible, section storyboard, signature moment, responsive rules, and QA checklist.

5. **Output**
   Deliver one of three outputs: concept direction, visual variants, or production build plan.

### Output

The tiny workflow should produce:

- One-sentence positioning
- Premium mode
- Source scorecard
- Art direction summary
- Section storyboard
- Signature moment
- Anti-slop rules
- Next action

## 2. Input Quality Score

Bad inputs should not lead the direction. Every image, link, transcript, or reference gets scored before it influences the build packet.

Score each from 0 to 5.

| Dimension | Question |
|---|---|
| Relevance | Does this source actually fit the business and audience? |
| Clarity | Is the visual/content signal easy to understand? |
| Originality | Does it avoid generic AI/template patterns? |
| Brand fit | Does it support the intended premium feeling? |
| Conversion usefulness | Does it help sell, explain, prove, or convert? |
| Technical usefulness | Can it be translated into real responsive UI? |
| Rights / permission | Can we use it directly, or only as inspiration? |
| Freshness | Does it feel current, or is it stale? |

### Role Rules

- **Lead source**: average score 4.0+ and strong brand fit.
- **Support source**: average score 3.0+.
- **Constraint source**: source must be preserved, even if visually weak.
- **Rejection source**: useful mainly for what to avoid.
- **Do not use**: average below 2.5 unless there is a specific reason.

### Hard Stops

A source cannot lead if:

- We do not have rights to clone it directly.
- It conflicts with the target audience.
- It is attractive but unrelated to the offer.
- It would make the result look like another brand.
- It breaks mobile or performance in obvious ways.

## 3. Client-Facing Intake Form

The intake form should be simple enough for non-designers.

Core questions:

1. What is the business or project name?
2. What do you sell or offer?
3. Who is this for?
4. What should visitors do?
5. What should the site make people feel?
6. Share 1 to 5 links you like.
7. Share 1 to 5 links you dislike.
8. Upload images, screenshots, logos, or product visuals.
9. What must the design avoid?
10. Do you own the linked sites/assets, or are they inspiration only?

The detailed version lives in [client-intake-form.md](client-intake-form.md).

## 4. Before/After Proof Loop

The system must prove it consistently beats normal AI website output.

### Test Method

For each case study:

1. Save the original input brief.
2. Generate a baseline "normal AI website" from the same brief.
3. Generate the Premium Site OS direction/build.
4. Compare both outputs using the same rubric.
5. Capture screenshots and notes.
6. Record what improved and what still failed.

### Rubric

Score 1 to 5:

- Strategic fit
- Originality
- Typography
- Layout and spacing
- Motion purpose
- Mobile readiness
- Conversion clarity
- Performance readiness
- Accessibility readiness
- Anti-slop score

### Passing Standard

Premium Site OS should beat the baseline by at least 20% overall and have no score below 3 in mobile, accessibility, performance, or conversion clarity.

4. Local service business
5. Product screenshot-led startup site
6. Wellness/calm commerce page
7. Technical command-center site
8. Portfolio/agency site
9. Existing ugly site to premium redesign
10. Transcript-methodology inspired concept

Each example should include:

- Inputs
- Source scorecard
- Creative direction
- Art bible
- Before/after screenshots
- QA score
- Lessons learned
- Taste memory entries

## 6. Cost and Time Modes

Use modes to prevent every job from becoming huge.

### Quick Concept

Time: 30 to 60 minutes.

Best for:

- Early idea exploration
- Sales calls
- Direction testing

Output:

- Source read
- Premium mode
- Art direction summary
- Section storyboard
- Signature moment idea

### Premium Concept

Time: 2 to 4 hours.

Best for:

- Serious client pitch
- Design direction before build
- Comparing variants

Output:

- Full source scorecard
- Trend/taste scan
- Art bible
- 2 to 3 high-leverage variants
- One-shot build packet
- QA rubric

### Production-Ready Build

Time: 1 to 5 days depending on scope.

Best for:

- Client delivery
- Launch-ready landing page
- Rebuild of an owned site

Output:

- Build packet
- Implemented site
- Responsive checks
- Performance pass
- Accessibility pass
- Anti-slop pass
- Deployment handoff

## 7. Taste Memory

Taste memory is the system's learning layer.

It records:

- What worked
- What failed
- What became stale
- What clients liked
- What users responded to
- What trends aged badly
- What implementation traps repeated

### Taste Memory Entry

```markdown
## [Pattern Name]

Status: Worked | Failed | Stale | Needs Retest
Date observed:
Source project:
Audience:
Premium mode:
Pattern:
Why it worked or failed:
Evidence:
When to use:
When to avoid:
Refresh date:
```

### Freshness Rules

- Review trend-based entries after 90 days.
- Review implementation patterns after 180 days.
- Promote repeated wins into reusable patterns.
- Demote overused patterns into anti-slop warnings.

## Operating Principle

Do not expose the whole machine to the client.

The client sees:

> Send links, images, and a goal. Get a premium direction.

The system handles:

> Source scoring, trend scan, art bible, variants, proof loop, QA, and taste memory.
